应用场景讨论

/ultrareview 架构（5/1）：云端并行 agent fleet，每个发现独立 agent 二次验证，官方误报率 <1%。本地 /review 单 pass，ultrareview 多 agent + 验证层。官方称 5–10 分钟；社区实测大 diff 15–20 分钟。「Max 订阅是起点而非终点」——首个 per-review 额外计费的开发者工具 SKU。

/ultrareview bug（4/28，GitHub Issues #54229，open 状态）：~6.6MB diff / 23 个 Python 文件 monorepo 中 sandbox 启动后无 agent activity，30 分钟 timeout 失败但免费 run 额度已扣除。2 次连续失败模式一致。

Opus 4.7 长 session 迭代问题（4/27，GitHub Issues #53958）：claude-opus-4-7[1m] 6 小时 session 内经 8 轮 Codex review 才完成一个 PR；模型「修变体不修类」——同一架构缺陷反复发现。用户请求模型主动调取历史 feedback memory + 自我报告「我在修变体」信号。

评价

Boris Cherny（CC Lead）：Opus 4.7 「更 agentic、更精确、长时运行更好、跨 session 上下文保持更好、处理模糊性更好」（X @bcherny）；驳斥 regression 说法，但 Anthropic PM 同期承认「正在快速推进内部 tuning」。

xhigh 隐性成本（llmtest.io 4/21）：xhigh ≈ high 的 2x token 用量；新 tokenizer 再叠 1.0–1.35x 膨胀；API 用户账单 +2–2.7x；订阅用户看不到账单但用量限制更快到顶。5/5 /ultrareview 免费到期后订阅用户首次直接感受 per-review 计费。

双工具 workflow 成主流：CC（Opus 4.7 跑 xhigh）做 implementation + UI；Codex（GPT-5.5）做 review / QA / investigation；Gemini CLI 做大 context 批量扫描。
